# Visual Auditor Self-Learning System — Specification

**Date:** 10 April 2026
**Status:** Proposed

---

## 1. Problem

The Visual Auditor currently operates as a stateless critic — it reviews slides, flags issues, and forgets everything. Each new deck starts from zero. There are two missed opportunities:

1. **Auditor suggestions could drive automatic slide rework** — when the auditor says "use an infographic instead of a table", the system should be able to act on that, not just log it.

2. **User feedback and auditor findings could compound** — patterns like "tables with <6 rows should be icon_stat" or "Aigis always uses cream background" should be learned and applied to future decks without re-discovery.

## 2. Architecture: Three Components

```
┌─────────────────────────────────────────────────────┐
│            VISUAL AUDITOR SELF-LEARNING              │
│                                                      │
│  A. SUGGESTION → ACTION ENGINE                       │
│     Auditor finding → classify → auto-rework slide   │
│                                                      │
│  B. PATTERN MEMORY (per-brand)                       │
│     Learned design patterns stored in YAML            │
│     Fed into DesignAdvisor + Auditor prompts          │
│                                                      │
│  C. USER FEEDBACK LOOP                               │
│     Accept/reject/modify → update pattern memory      │
│     Approval rate drives auto-apply thresholds        │
└─────────────────────────────────────────────────────┘
```

## 3. Component A: Suggestion → Action Engine

### Current state
The auditor returns findings like:
```json
{"severity": "warn", "message": "INFOGRAPHIC > TABLE: This 4-row pricing table would be more effective as an icon_stat layout"}
```
Currently, this warning is printed and ignored.

### Proposed: actionable suggestion classification

Each auditor finding gets classified into one of:

| Category | Action | Example |
|----------|--------|---------|
| **layout_change** | Re-pick slide type via DesignAdvisor | "Table → icon_stat", "Table → feature_grid" |
| **content_trim** | Reduce/shorten content in slide_fixer | "Too many bullets", "Text too long" |
| **spacing_fix** | Modify Typst source spacing/sizing | "Whitespace below cards", "Chart too small" |
| **brand_fix** | Adjust colours/fonts in renderer | "Icons should use single accent" |
| **data_viz** | Re-render chart with modifications | "Highlight client position", "Add axis labels" |
| **structural** | Template-level change needed | "Card heights unequal" — requires renderer fix |
| **not_actionable** | Log only | "Weak visual hierarchy" — subjective |

### Implementation

New function in `slide_fixer.py`:

```python
def rework_from_suggestions(
    slides: list[dict],
    suggestions: list[AuditWarning],
    advisor: DesignAdvisor,
    brand: str,
) -> tuple[list[dict], list[dict]]:
    """
    Apply auditor suggestions as slide reworks.
    
    For layout_change suggestions:
    1. Extract the affected slide's original section data
    2. Call DesignAdvisor with a targeted prompt:
       "The current slide uses {current_type} but the auditor recommends
        {suggested_type}. Redesign this slide as {suggested_type} using
        the same data."
    3. Replace the slide spec with the redesigned version
    
    Returns (reworked_slides, change_log)
    """
```

For `layout_change` actions specifically:
- Parse the suggestion message for the recommended slide type (regex: `"as an? (\w+) layout"`, `"use (\w+) instead"`)
- Feed the original section data + the suggested type back to DesignAdvisor with `slide_mode: "guided"` and the suggested `slide_type` pre-set
- The LLM fills the data fields for the new type while preserving the content

This means the outer visual audit loop becomes:
```
Compile → Audit → Classify suggestions → Auto-rework actionable ones → Recompile → Re-audit
```

### Example flow

Auditor says: *"INFOGRAPHIC > TABLE: This 4-row pricing table would be more effective as an icon_stat layout"*

1. Classify: `layout_change` → target type `icon_stat`
2. Extract slide 12's data: `{headers: ["Tier", "Model", "Price"], rows: [...]}`
3. Call DesignAdvisor: "Redesign this as icon_stat with these 4 data points"
4. LLM returns: `{slide_type: "icon_stat", data: {stats: [{value: "£11K", label: "ANALYST", ...}]}}`
5. Replace slide 12 in the deck
6. Recompile and re-audit

## 4. Component B: Pattern Memory (per-brand)

### Purpose
Store learned design patterns so they persist across deck generations. Each brand accumulates its own design knowledge over time.

### Storage: `~/.config/inkline/patterns/{brand_name}.yaml`

```yaml
# Learned design patterns for the aigis brand
# Auto-generated by Visual Auditor + user feedback
brand: aigis
version: 3
last_updated: 2026-04-10

patterns:
  - id: p001
    category: layout_preference
    rule: "Tables with ≤5 rows of key-value data should use icon_stat or kpi_strip"
    confidence: 0.85
    source: auditor_suggestion
    approved_by_user: true
    applied_count: 4
    
  - id: p002
    category: brand_rule
    rule: "Aigis always uses cream background (#E7E7E3), never pure white"
    confidence: 1.0
    source: user_correction
    approved_by_user: true
    
  - id: p003
    category: chart_style
    rule: "Competitive landscape scatter must highlight Aigis with glow ring + boxed label"
    confidence: 1.0
    source: user_correction
    applied_count: 3
    
  - id: p004
    category: layout_preference
    rule: "Exit strategy should use timeline with increasing bubble sizes, not plain table"
    confidence: 0.90
    source: user_feedback
    approved_by_user: true

  - id: p005
    category: typography
    rule: "Title slide company name must be uppercase bold ≥72pt"
    confidence: 1.0
    source: user_correction
```

### Integration

**Into DesignAdvisor:** When designing a deck for brand X, load `patterns/{brand}.yaml` and inject the rules into the system prompt:

```python
def _build_system_prompt(self, ...):
    # ... existing playbook loading ...
    
    # Load brand-specific learned patterns
    patterns = load_brand_patterns(self.brand)
    if patterns:
        parts.append("=" * 60)
        parts.append("LEARNED PATTERNS FOR THIS BRAND")
        parts.append("=" * 60)
        for p in patterns:
            parts.append(f"- {p['rule']} (confidence: {p['confidence']})")
```

**Into Visual Auditor:** Patterns also feed into the auditor's system prompt so it doesn't flag things that have already been resolved:

```python
def _build_visual_audit_system():
    patterns = load_brand_patterns(current_brand)
    # Add: "Known brand rules — do NOT flag these as issues:"
    # e.g., "cream background is intentional, not a brand violation"
```

### Pattern lifecycle

```
1. Auditor flags an issue → suggestion logged
2. System auto-applies fix (if actionable)
3. If fix works (auditor passes on re-audit):
   → Pattern recorded with confidence 0.5
4. If user reviews and approves:
   → Confidence bumped to 0.85
5. If pattern applied successfully 3+ times:
   → Confidence bumped to 0.95 (auto-apply threshold)
6. If user rejects:
   → Pattern marked as rejected, not applied again
```

## 5. Component C: User Feedback Loop

### Feedback capture points

1. **Post-generation review:** After deck ships, user reviews in their PDF viewer. Feedback captured via CLI:
   ```
   inkline feedback aigis-investor-v6.pdf
   > Slide 3: cards should be equal height [structural]
   > Slide 15: bubble timeline is great, keep this pattern [positive]
   > Slide 9: market table should be a bar chart [layout_change]
   ```

2. **Accept/reject cycle:** When the system proposes a rework (from Component A), user can approve or reject:
   ```
   [REWORK PROPOSAL] Slide 12: table → icon_stat for pricing tiers
   Accept? (y/n/m for modify): y
   → Pattern saved: "Pricing data with ≤4 tiers should use icon_stat"
   ```

3. **Implicit feedback:** If the user manually edits a slide's section data and re-runs, the system detects the delta:
   - User changed `slide_type` from table to icon_stat → pattern learned
   - User changed bubble sizes → pattern updated
   - User changed text → no pattern (content-specific)

New module: `src/inkline/intelligence/feedback.py`


### Auto-apply thresholds

| Confidence | Behaviour |
|-----------|-----------|
| < 0.5 | Suggestion only (logged, not applied) |
| 0.5 - 0.8 | Proposed to user for approval |
| 0.8 - 0.95 | Applied automatically, user notified |
| ≥ 0.95 | Applied silently (proven pattern) |

This mirrors Aigis's own self-learning system (proposals → human review → auto-apply after 80% approval rate).

## 7. Immediate impact

With this system:
- First deck for a new brand: full iteration cycle, many corrections
- Second deck: 60% fewer corrections (learned patterns applied)
- Fifth deck: 90% fewer corrections (patterns at auto-apply confidence)
- User feedback compounds: every correction makes future decks better

The exit strategy bubble timeline, the Aigis scatter highlight, the cream background, the shield placement — all of these would be learned patterns that apply automatically on future Aigis decks without being told again.
